# AI 智能视频监控系统发布与部署方案

> 本文档用于回答一个很实际的问题：这套系统已经在当前开发机上跑通了，后续如何像工业项目一样，稳定地复制到许多其他同型号机器上，而不是每台机器都手工补依赖、改配置、反复踩坑。

---

## 一、问题本质

当前这套系统不是一个单一程序，而是一整套由多个服务和平台依赖组成的工业级系统，包含：

- Vue 前端
- Go 后端
- Python 算法调度服务
- C++ 推理服务
- ZLMediaKit
- SQLite 数据库
- 模型文件、标签文件、音频文件
- Rockchip 平台相关依赖，如 `mpp`、`rknn runtime`、`rga`、`ffmpeg-rk` 等

开发阶段常见的做法是：

- 缺什么就临时安装什么
- 路径不统一时手工改配置
- 某个依赖不兼容时现场修
- 在开发机上边开发边把环境“养起来”

这种方式在单机开发时可以接受，但到了交付和批量部署阶段就会遇到问题：

- 无法保证每台机器环境完全一致
- 很难复现“这台为什么能跑、那台为什么不能跑”
- 人工安装成本很高
- 后续升级和回滚都很困难

所以工业界的核心思路不是“把开发机目录直接拷过去”，而是把整个系统拆成几层，分别标准化。

---

## 二、工业界一般怎么做

对于你这种嵌入式工控项目，工业界通常不会在每台机器上重复手工搭环境，而是采用下面这种思路：

### 1. 基础运行环境单独固化

把和硬件平台强相关、安装复杂但变动不频繁的内容，做成统一的基础环境，例如：

- Linux 系统本身
- Rockchip 相关驱动和运行时
- `mpp`
- `rga`
- `rknn runtime`
- `ffmpeg-rk`
- `ZLMediaKit`
- Java 运行时
- `jemalloc`
- 其他系统库和工具链

这一层最适合做成：

- 标准系统镜像
或
- 固定版本的基础软件包集合

### 2. 业务应用单独发版

把会频繁迭代的部分，单独做成版本化的发布包，例如：

- Go 后端可执行文件
- Python 算法调度服务
- C++ 推理服务二进制
- 前端构建后的 `dist`
- 模型文件
- 音频文件
- 初始化 SQL
- systemd 服务文件
- 配置模板

这样每次升级业务功能时，不需要重新手工安装一遍系统依赖。

### 3. 配置与程序分离

工业项目通常不会把路径、端口、模型位置、上传地址、设备编号、数据库路径等写死在代码里，而是集中放到配置文件或环境变量中。

这样做的好处是：

- 程序包可以通用
- 不同机器只改配置，不改代码
- 便于批量部署

### 4. 服务由 systemd 管理

正式项目不会依赖人工按顺序敲命令启动，而是交给 `systemd` 统一管理：

- 开机自启
- 按依赖顺序启动
- 自动重启
- 查看日志
- 统一运维

### 5. 批量复制靠镜像、安装包和自动化工具

当机器数量多起来后，常见方法是：

- 先刷统一基础镜像
- 再下发应用发布包
- 通过脚本、`ssh`、`scp`、`Ansible` 等工具批量安装或升级

所以关键不是“每台机器怎么手工装”，而是“如何把手工过程沉淀成可重复执行的标准流程”。

## 三、对本项目最合适的发布思路

结合当前项目架构，最推荐采用以下两层方案：

### 第一层：基础镜像

制作一份 RK3576 工控机的标准基础镜像，里面预装：

- Linux 系统
- Rockchip 运行时和相关库
- `mpp`
- `rga`
- `rknn runtime`
- `ffmpeg-rk`
- `ZLMediaKit`

例如预先建立：

### 第二层：应用发布包

把业务系统做成一个清晰的版本化发布包，只包含会变化的内容。

│   ├── ai-monitor-backend
│   └── infer_server
├── python/
│   ├── app/
│   └── venv/ 或 wheels/
├── frontend/
│   └── dist/
├── models/
│   ├── yolo11n-rk3576.rknn
│   └── yolo11n-labels.txt
├── audio/
├── sql/
│   ├── schema.sql
│   └── seed.sql
├── config/
│   ├── app.env
│   ├── config.ini
│   └── server.json
├── systemd/
│   ├── zlm.service
│   ├── infer.service
│   ├── ai-monitor-python.service

这样以后往其他机器部署时，就不是手工拷目录和调试环境，而是：

1. 刷基础镜像
2. 拷贝发布包
3. 执行安装脚本
4. 修改少量机器专属配置
5. 启动 systemd 服务

## 四、工业界常见的几种部署方式

### 1. 批量刷系统镜像

这是嵌入式和工控行业最常见、最稳的一种方案。

适用场景：

- 机器型号一致
- 系统版本一致
- 硬件依赖重
- 环境复杂

做法：

- 先把开发机整理成标准母盘
- 导出为镜像
- 用量产工具、烧录工具、Clonezilla、`dd` 等方式批量复制
- 首次启动后再执行一次初始化脚本

优点：

- 复制结果最稳定
- 最接近“开发机能跑，目标机也能跑”
- 对 `rknn`、`mpp`、`ffmpeg-rk` 这类强平台依赖特别友好

缺点：

- 升级不够灵活
- 如果只是改应用，整机镜像就显得偏重

### 2. 基础镜像 + 应用安装包

这是最推荐的方式，也是很多工业项目的实际做法。

做法：

- 先统一刷基础镜像
- 再下发某个版本的应用发布包
- 后续只升级应用层

优点：

- 环境稳定
- 升级方便
- 便于回滚
- 便于做版本管理

### 3. 局域网批量自动化部署

适合设备数量变多以后使用。

典型工具：

- `Ansible`
- `SaltStack`
- 自己写的 `ssh + scp + shell`

做法：

- 控制机统一保存发布包
- 通过脚本批量传输到目标机器
- 远程执行 `install.sh` 或 `upgrade.sh`
- 收集结果、检查服务状态

### 4. Docker 容器化

工业界也会使用容器，但在你这个项目里需要谨慎。

原因是：

- Go 后端、前端、Python 服务容器化相对容易
- 但 `infer_server`、`ffmpeg-rk`、`rknn`、`mpp`、`rga`、设备节点访问、媒体性能优化等都强依赖宿主机环境

所以对当前阶段更实际的建议是：

先做：

- 原生部署
- systemd 管理
- 基础镜像 + 应用包

以后如果项目成熟，再考虑将部分纯业务服务容器化。

## 五、你这个项目最应该建立的发布思维

不要再把系统看成“开发目录的集合”，而应该拆成下面五类资产。

### 1. 系统依赖

不常变化，但安装复杂、和平台强相关：

- `mpp`
- `rga`
- `rknn runtime`
- `ffmpeg-rk`
- `ZLMediaKit`
- Java
- `jemalloc`

这些应当放入基础镜像，或者做成固定版本离线安装包。

### 2. 应用程序

经常变化、需要频繁发版：

- Go 后端
- Python 算法服务
- C++ 推理服务
- 前端 `dist`

这些应当由构建脚本统一生成，并放入版本化发布包。

### 3. 配置

例如：

- 端口
- 数据库路径
- 模型目录
- 快照目录
- ZLM 配置
- 上传地址
- 设备 ID
- 音频设备参数

这些应当集中管理，不要散落在代码和临时脚本中。

### 4. 数据

例如：

- `aimonitor.db`
- 摄像头信息
- 任务信息
- 告警记录
- 系统设置

工业项目通常不会把开发机上的运行中数据库原样拷到所有目标机器，而是：

- 用 `schema.sql` 初始化结构
- 用 `seed.sql` 导入算法字典、默认参数、默认系统配置
- 现场机器自己的摄像头、任务、设备号单独配置

也就是说：

- 通用默认数据可以随发布包下发
- 现场业务数据要和开发测试数据分开

### 5. 服务管理

系统正式上线后，必须让服务自动化运行，而不是依赖人工启动。

适合拆成如下几个服务：

- `zlm.service`
- `infer.service`
- `ai-monitor-python.service`
- `ai-monitor-backend.service`

前端生产环境建议由 Go 后端直接托管 `dist`，而不是继续使用 Vite 开发服务器。

## 六、针对当前项目的建议路线

下面这条路线最适合当前阶段，成本不高，但能快速从“开发机工程”过渡到“可交付工程”。

### 第一步：先做到可重复安装

目标不是批量部署，而是先确保你自己能在一台全新同型号机器上，完整重装成功一次。

要具备以下内容：

- 一份基础依赖安装清单
- 一份应用安装脚本
- 一份数据库初始化脚本
- 一份配置模板
- 一份服务启动方案

只要一台干净机器能从零装成功，就已经迈出了最关键的一步。

### 第二步：把基础环境固化成镜像

当第一步验证通过后，把那些不常变、安装繁琐的内容写进基础镜像：

- Rockchip 运行时
- `ffmpeg-rk`
- `ZLMediaKit`
- Java
- 目录结构
- systemd 基础服务框架

以后新机器部署时，只需要：

1. 刷镜像
2. 拷贝应用发布包
3. 执行安装脚本
4. 修改设备专属配置

### 第三步：再考虑批量自动化

当设备数量变多以后，再逐步引入：

- `Ansible`
- 局域网批量升级脚本
- 版本检查和自动回滚机制

这一步不是最急的，重点是先把前两步做扎实。

## 七、对当前项目的具体落地建议

下面按你现在这套目录结构，给出一份几乎可以照着实施的方案。

### 1. 先整理“哪些内容进基础镜像”

建议基础镜像包含：

- Linux 系统本身
- Rockchip 驱动和相关平台库
- `mpp`
- `rga`
- `rknn runtime`
- `ffmpeg-rk`
- `ZLMediaKit`
- Java runtime
- `jemalloc`
- `sqlite3`
- `systemd`

同时预创建目录：

- `/opt/ai-monitor/`
- `/opt/ai-monitor/bin/`
- `/opt/ai-monitor/python/`
- `/opt/ai-monitor/frontend/`
- `/opt/ai-monitor/models/`
- `/opt/ai-monitor/audio/`
- `/etc/ai-monitor/`
- `/var/lib/ai-monitor/`
- `/var/log/ai-monitor/`

其中建议：

- 程序放 `/opt/ai-monitor/`
- 配置放 `/etc/ai-monitor/`
- 数据库和运行数据放 `/var/lib/ai-monitor/`
- 日志放 `/var/log/ai-monitor/`

这样会比全部放在 `/home/hzhy/` 下更像正式产品，也更利于部署。

### 2. 明确“哪些内容进应用发布包”

建议应用发布包包含：

- Go 后端编译产物
- C++ 推理服务编译产物
- Python 算法服务源码
- Python 虚拟环境或离线 wheels
- Vue 前端构建后的 `dist`
- 模型文件和标签文件
- 音频文件
- SQL 初始化文件
- 配置模板
- systemd 单元文件
- 安装/升级/检查脚本

### 3. 建立统一构建流程

建议不要再用“开发机上哪个目录刚好能跑”作为发布依据，而是固定构建流程。

例如：

- Go 后端：固定使用同一版本 Go 编译，输出单一二进制
- 前端：统一执行 `npm run build`
- Python：固定生成 `venv` 或离线 wheel 包
- C++ 推理服务：固定编译参数和依赖版本

最终把所有产物集中收集到一个发布目录里。

### 4. 统一配置文件

建议把散落的配置集中起来，形成一套正式配置文件，例如：

- `/etc/ai-monitor/app.env`
- `/etc/ai-monitor/zlm.ini`
- `/etc/ai-monitor/infer-server.json`
- `/etc/ai-monitor/backend.yaml`
- `/etc/ai-monitor/python.yaml`

配置项建议至少覆盖：

- 数据库路径
- 快照路径
- 模型路径
- 音频路径
- ZLM 地址和 secret
- 推理服务地址
- Python 服务地址
- Go 服务监听地址
- 上传服务器地址
- 设备 ID
- 日志路径

### 5. 数据库要区分“结构、默认数据、现场数据”

建议把数据库处理拆成三部分：

- `schema.sql`：只负责建表
- `seed.sql`：负责初始化算法字典、默认模型、默认系统配置
- 现场配置导入文件：负责各机器不同的摄像头、任务、设备编号

不要直接把开发机上的 `aimonitor.db` 原封不动复制到所有机器上。

因为开发数据库里通常混有：

- 测试摄像头
- 临时任务
- 历史告警
- 调试配置
- 不适用于现场的路径和地址

### 6. 采用 systemd 统一管理服务

建议建立如下服务：

- `zlm.service`
- `infer.service`
- `ai-monitor-python.service`
- `ai-monitor-backend.service`

依赖关系大致如下：

- `infer.service` 依赖网络和基础环境
- `ai-monitor-python.service` 依赖 `infer.service`
- `ai-monitor-backend.service` 依赖 `ai-monitor-python.service`

如果前端由 Go 托管，则不需要单独前端服务。

这样做的好处：

- 开机自动拉起
- 异常自动重启
- 统一日志查看
- 现场运维难度低

### 7. 编写安装脚本和升级脚本

建议至少有两个脚本：

#### `install.sh`

负责：

- 检查系统依赖是否齐全
- 创建目录
- 拷贝程序文件
- 拷贝模型、音频、前端文件
- 安装配置模板
- 初始化数据库
- 安装 systemd 单元
- `daemon-reload`
- 启用并启动服务
- 执行健康检查

#### `upgrade.sh`

负责：

- 停止业务服务
- 备份旧版本二进制、配置和数据库
- 替换新版本程序和资源
- 执行数据库升级脚本
- 重启服务
- 健康检查
- 失败时支持回滚

### 8. 做版本清单和发布记录

每次发版都建议带上：

- `VERSION`
- `manifest.json`
- `release-notes.md`

至少记录：

- backend 版本
- python 版本
- infer_server 版本
- frontend 版本
- model 版本
- schema 版本
- ZLM 版本
- ffmpeg-rk 版本

这样后续定位问题时，能知道每台机器到底装的是什么。

## 八、为什么不建议继续按现在这种方式扩散部署

如果继续沿用“开发机缺什么装什么，哪台机器有问题就手工修哪台”的方式，会带来这些问题：

- 机器间环境漂移
- 问题无法复现
- 发布结果不可追踪
- 新人很难接手
- 现场交付耗时且风险高
- 后续升级越来越难

项目刚开始时这样很正常，但只要准备进入多机部署阶段，就必须尽快把这部分工程化。

---

## 九、对当前阶段最实用的建议

对你现在来说，不必一开始就上最复杂的 DevOps 体系，也不必强行容器化全部服务。最实际的路径是：

1. 先整理一份完整依赖清单
2. 先完成一次“全新机器从零安装成功”的验证
3. 把安装过程沉淀成 `install.sh`
4. 把服务收敛到 `systemd`
5. 把基础依赖做进统一镜像
6. 再把应用做成版本化发布包
7. 机器多了之后再考虑 `Ansible` 或内网批量升级

这条路线非常符合嵌入式和工业项目的真实推进节奏。

## 十、一句话总结

对于这套 RK3576 工控机上的 AI 视频监控系统，最适合的工业化发布方式是：

**先把强平台依赖做成统一基础镜像，再把 Go、Python、C++、前端、模型、配置、SQL 和 systemd 服务文件做成版本化应用发布包，通过安装脚本和自动化工具批量部署，而不是在每台机器上手工补环境。**
